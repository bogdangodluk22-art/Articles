// 01 — OVERVIEW

What the platform does

Four areas that together cover the full cycle — from niche research to a concrete action plan for the next 30, 60 or 90 days.

📊
Channel and competitor analytics
The system scrapes YouTube, Instagram and TikTok on its own via Apify: subscribers, average views, ER, top videos. In parallel — reconnaissance of up to 3 competitors in your niche, surfacing open formats and weak spots.
💡
Idea generation with context
A content plan for 10–30 videos is generated by GPT-4o using the data from the previous floors — channel, competitors, trends. Each idea: title, hook, format (long/Shorts), runtime. Without Supabase context — it generates by niche.
📈
Real-time trend analysis
Tavily searches each selected platform in parallel — what's gaining momentum right now, what's already burned out. GPT-4o consolidates the data into three blocks: hot trends, dead topics, and a concrete tip on "what to film right now".
🚀
Growth strategy with full context
The final tool gathers the saved results of all 4 previous floors from Supabase and builds a personal strategy for 30/60/90 days: phases, concrete actions, KPIs with numbers. It works even without prior data — based on your niche and goal.

// 03 — ARCHITECTURE

How it works — the full flow

From the first request to a saved result in the dashboard. Seven stages, each one a separate module with a clear responsibility.

01
🖥️ Frontend — one HTML file, 5000+ lines
The entire frontend is pure HTML/CSS/JS in a single file. A skyscraper metaphor: the floors are CSS scroll sections, with an elevator animation between them via JS. Each floor has a unique background room, a TV in the center and an ON/OFF remote. When the remote is pressed, spotlights from all 4 corners light up the TV (CSS conic-gradient + spotlight-pulse animation). The tool window "flies out" of the TV. Navigation — a sticky strip at the top with auto-scroll, and up/down arrows between floors. On any transition — user_state.tool is updated in Supabase.
💡 Mobile lock: a CSS media query + a JS overlay "Available on desktop only". Design system: Bebas Neue + Syne + Space Mono, a gold cursor (cursor:none + custom SVG, z-index 999999).
Pure HTML/CSS/JS CSS Animations Scroll Sections Babylon.js 3D Supabase Auth
02
🔐 Authorization — phone + password → Supabase
Custom authorization with no third-party OAuth. The form collects a phone number and password → SHA-256 hash → POST to the Supabase users table. On a successful login, localStorage saves user.id (a real UUID from Supabase). This user_id is the key to every operation: Osiris dialogues, user_state, saved tool_results. UPSERT user_state: a POST with Prefer: resolution=merge-duplicates — both scenarios (new and existing user) through a single request.
💡 A critical fix: the old user_id format (site_abc123_timestamp) generated new records for every session. After the fix — a real UUID from Supabase, one row per user.
Supabase users table SHA-256 password hash localStorage UUID UPSERT user_state
03
⚡ n8n Webhook — request routing
Each floor has its own webhook endpoint on n8n (self-hosted Hostinger VPS). The standard pattern: Webhook → Validate Input (Code node, throws an error if required fields are missing) → Logic/Scraping → GPT-4o → Parse Response (strips ```json wrappers, returns a fallback if the JSON is invalid) → Respond (200 + Access-Control-Allow-Origin: *). Separate workflows: save-result (/gotrends-save-result) and get-results (/gotrends-get-results).
💡 All webhooks respond with Access-Control-Allow-Origin: * — CORS doesn't block requests from the frontend. responseMode: responseNode gives full control over the response.
n8n self-hosted VPS POST Webhooks Validate → Logic → Respond CORS headers
04
🔎 Data collection — Apify + Tavily
Floors 1 and 2: Apify actors for YouTube Data Scraper, Instagram Profile Scraper, TikTok Profile Scraper. Floor 2 — splitInBatches processes the competitors sequentially, saving to a static array (not a Code-Merge that wipes the data). Floors 2 and 4: Tavily with search_depth: advanced, max_results: 6, include_answer: true. Floor 4 — Split Platforms → a parallel Tavily for each platform → Merge Results into a single text block.
💡 Platform auto-detection by URL: youtube.com → YouTube Apify actor, instagram.com → IG actor, tiktok.com → TikTok actor. Both a URL and an @handle are supported.
Apify YouTube/IG/TikTok Tavily Advanced Search splitInBatches Loop Parallel Requests
05
🧠 GPT-4o — analysis and generation
Each floor has its own system prompt and JSON response schema. Floor 3 (content plan): temperature 0.85, max_tokens 3000. Floor 4 (trends): temperature 0.4, max_tokens 600 — a strict structure. Floor 5 (strategy): temperature 0.7, max_tokens 2000, a context block with F1–F4 data. Osiris AI Agent: GPT-4o via the n8n AI Agent node with RAG (match_documents, pgvector) + direct reading of tool_results (MAX_TOOL_CHARS: 3000, MAX_HISTORY_CHARS: 2000).
💡 All prompts require "ONLY valid JSON with no text before or after". Parse Response always strips ```json wrappers and has a try/catch with a fallback object.
GPT-4o JSON-only responses RAG pgvector Context Injection Fallback Parsing
06
💾 Saving results — tool_results → Dashboard
After receiving a result, the user clicks "Save". The frontend sends a POST to /gotrends-save-result: {user_id, tool, platform, result}. The workflow validates it (an allowedTools list), builds the payload → POST to the Supabase tool_results table with Prefer: return=representation. The dashboard (/gotrends-get-results) loads all of the user's records, groups them by tool (taking the latest + history with a preview), and returns data + history objects. showFNResult → lupa-modal → saveBtn — the standard pattern on every floor.
💡 tool_results.result is a JSONB field. Each tool stores its own schema. The get-results preview line pulls a key field depending on the tool: channel_analysis → channel.name, content_plan → ideas.length + niche, and so on.
tool_results JSONB Supabase POST + UPSERT showFNResult pattern Dashboard Preview
07
📱 Telegram — Bot + Web App + Osiris Page
Osiris Telegram Bot workflow: /start → sendPhoto → Wait (3s) → editMessageMedia (GIF + welcome + 3 menu buttons) via HTTP Request (not the native TG node — it doesn't support editMessageMedia). 3 buttons: open the website, launch the Web App, activate Osiris. Web App: the same osiris-webapp.html on the Hostinger VPS, transparent background, Osiris expanded right away, user_id = Telegram.WebApp.initDataUnsafe.user.id. Deploy: locally → push to GitHub → curl the raw URL onto the VPS. The 5 tool buttons set user_state.tool in Supabase.
💡 HTTP Request instead of the Telegram node for editMessageMedia — a critical detail. The IF node in n8n: ={{ $json.id }} with is empty (not $json[0]?.id — n8n auto-unwraps arrays).
Telegram Bot API Web App (Mini App) editMessageMedia user_state.tool GitHub → VPS Deploy
// 04 — OSIRIS AI

The platform's 3D assistant

Osiris is the brand's third hero. A smart machine with an Egyptian Eye logo, a Babylon.js 3D model and a GPT-4o brain. Present on every floor and aware of the context the user is in.

😴
COLLAPSED
Only the Eye avatar is visible (an Egyptian SVG with a gold pulse animation). The indicator button is red. Takes up minimal space in the corner of the screen.
🔄
EXPANDED — INACTIVE
The base.glb 3D model — a constant slow rotation (spin). Waiting for activation. The ACTIVATE button is visible. Babylon.js renders the GLB from GitHub Pages (bogdangodluk22-art/osiris-assets).
🟢
ACTIVE
Switches to breathing.glb (a breathing idle animation). A dialogue window appears with parseMarkdown() — rendering Markdown via innerHTML. The button turns green. Webhook payload: {message, source: 'site'|'webapp', phone: userId}.
⌨️
TYPING
While generating a response — switches to touch.glb (typing animation). Returns to breathing.glb after receiving the response from n8n.
🔴
DEACTIVATED
Returns to base.glb (spin). The button is red again. The tool in Supabase is reset to fallback or the floor's previous state.

// 07 — LAUNCH

What's needed to set it up

Five steps from zero to a fully live platform. All the infrastructure is self-hosted on a single VPS.

01
Supabase — database and authorization Create a project, deploy 4 tables (users, user_state, dialog_history, tool_results) + the documents table with pgvector for RAG. Enable the vector extension. Save the anon key and URL.
02
n8n on a Hostinger VPS — the orchestrator Install n8n via Docker or npm. Import the 10 JSON workflows. Configure the credentials: Supabase GoTrends (HTTP Header Auth with apikey), OpenAI (openAiApi), Apify (API key), Tavily (API key). Activate all the workflows.
03
Babylon.js 3D models — Osiris on GitHub Pages Upload 3 GLB files (base.glb, breathing.glb, touch.glb) to a GitHub Pages repository. Get the public URL for each model. Wire them into the frontend code — Babylon.js loads the GLB directly from the CDN.
04
Frontend deployment — HTML on the VPS + Telegram Web App Copy gotrends.html to the Hostinger VPS into the nginx/apache directory. Register a domain or use the VPS IP. For the Telegram Web App — a separate osiris-webapp.html with a transparent background. Deploy: locally → push to GitHub → curl the raw URL onto the VPS.
05
Telegram Bot + Osiris Memory (RAG) Create a bot via @BotFather. Set the webhook URL to n8n. Register the domain for the Web App via /setdomain. Upload the GoTrends knowledge base to the Google Drive "Osiris" folder — the Osiris Memory workflow auto-vectorizes it and loads it into pgvector on every file update.
